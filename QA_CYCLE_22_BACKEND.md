## Issues Found

### Medium

#### 1. `app/dependencies.py:102,109` — `extract_api_key` returns API keys in original case, causing uppercase API keys to fail authentication

**Location:** `app/dependencies.py:102–109`  
**Description:**
```python
def extract_api_key(request: Request, x_api_key: Optional[str] = None) -> Optional[str]:
    if x_api_key:
        return x_api_key
    auth = request.headers.get("Authorization", "")
    if auth.lower().startswith("bearer "):
        token = auth[7:].strip()
        token_lower = token.lower()
        if len(token) == 64 and all(c in "0123456789abcdef" for c in token_lower):
            return token
    return None
```

The function validates that the token is 64 hex characters using a lowercased copy (`token_lower`), but returns the **original case** (`token`). The raw API key is generated by `secrets.token_hex(32)`, which produces **lowercase** hex. `hash_key(api_key_val)` computes a SHA-256 hash of the returned string. Because the hash is case-sensitive, an API key presented in uppercase (e.g., via `X-API-Key: AABB...` or `Bearer AABB...`) will produce a different hash than the stored lowercase hash, causing `hmac.compare_digest` to fail and the authentication to be rejected with `401 Invalid or expired API key` even though the key is valid.

**Impact:** A valid API key sent in uppercase will always fail authentication. This is a latent functional bug that could break integrations where clients uppercase headers.
**Fix:** Normalize the API key to lowercase before returning:
```python
if x_api_key:
    return x_api_key.lower()
# ...
    return token.lower()
```

#### 4. `app/routers/billing.py:120-130` — Unreachable dead code after the payment-status gate

**Location:** `app/routers/billing.py:112–130`  
**Description:** The `subscribe` endpoint hardcodes `payment_status="pending"` when creating the `Subscription` object. The subsequent check `if sub.payment_status != "completed"` is therefore **always** `True`, causing the endpoint to raise `402 Payment Required` and return. The code block that updates `user.tier = body.tier` and commits the transaction (lines 121–130) is **never executed**.

```python
sub = Subscription(
    user_id=user.id,
    tier=body.tier,
    start_date=start_date,
    end_date=end_date,
    payment_status="pending",  # Always "pending"
)
db.add(sub)

if sub.payment_status != "completed":  # Always True
    await db.commit()
    await db.refresh(sub)
    raise HTTPException(
        status_code=status.HTTP_402_PAYMENT_REQUIRED,
        detail="Payment required",
    )

# The following code is unreachable:
user.tier = body.tier
...
```

**Impact:** Dead code that will never execute, misleading future maintainers and leaving the payment-integration placeholder incomplete.  
**Fix:** Either remove the unreachable block or restructure the logic so that `user.tier` is upgraded only when a payment processor confirms `payment_status = "completed"`.

#### 5. `app/models.py:191` — `Subscription` model missing `CheckConstraint("end_date >= start_date")`

**Location:** `app/models.py:180–212`


#### 6. `app/routers/ingest.py:86` — `ingest` endpoint does not verify station `status` before accepting readings

**Location:** `app/routers/ingest.py:60–77`  
**Description:** The `ingest` endpoint verifies that a station belongs to the user and is not soft-deleted (`deleted_at.is_(None)`), but it does **not** check that the station's `status` is `"active"`. A station in `"inactive"` or `"maintenance"` status can still receive sensor readings, which may be unintended.  
**Fix:** Add `SensorStation.status == "active"` to the ownership verification query:
```python
select(SensorStation).where(
    SensorStation.id.in_(station_ids),
    SensorStation.user_id == user.id,
    SensorStation.status == "active",
    SensorStation.deleted_at.is_(None),
)
```

#### 7. `app/routers/auth.py:305` — `/auth/me` endpoint lacks rate limiting

**Location:** `app/routers/auth.py:305–309`  
**Description:** The `/me` endpoint uses `get_current_user` directly and does not include `rate_limit_dependency`. While the endpoint is lightweight, an authenticated client could abuse it without any rate-limit enforcement. All other authenticated endpoints use `rate_limit_dependency`.  
**Fix:** Add `rate_limit_dependency` to the endpoint:

```python
@router.get("/me", response_model=StandardResponse)

    ...
```

#### 8. `app/auth.py:32` / `app/schemas.py:43` — Password length validation counts characters, not bytes; bcrypt truncates at 72 bytes

**Location:** `app/auth.py:32`, `app/schemas.py:43`  
**Description:** `validate_password` uses `len(password) < 8` (character count), and `UserRegisterRequest` caps `password` at `max_length=72` characters. `bcrypt` silently truncates input at **72 bytes**. If a user creates a password containing multi-byte characters (e.g., CJK characters at 3 bytes each, or emoji at 4 bytes each), a 72-character password could exceed 72 bytes and be silently truncated by `bcrypt`, effectively reducing the password entropy.  
**Fix:** Validate password length in bytes before hashing:
```python
if len(password.encode("utf-8")) > 72:
    raise ValueError("Password must be 72 bytes or less")
```
Alternatively, apply a pre-hash (e.g., HMAC-SHA256) to normalize the input length before passing to bcrypt, as recommended by passlib for passwords over 72 bytes.
