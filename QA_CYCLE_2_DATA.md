## High Issues

| # | Severity | File | Line | Description | Fix Suggestion |

| 7 | **High** | `data-pipeline/generators/reading_generator.py` | 173–226 | **Correlation logic is broken.** Temperature, PM2.5, and PM10 are not pre-computed per timestep before the sensor loop. Because `sensor_types` is sorted alphabetically (`air_quality` → `humidity` → `pm10` → `pm25` → `temperature`), `humidity` is generated with a **default temperature of 15.0°C** instead of the actual temperature for that timestamp. Similarly, `pm25` is **regenerated** after `pm10` and `air_quality` have already consumed a different value, destroying the physical correlation. | Pre-compute `temperature`, `pm25`, and `pm10` for every `ts` **before** the inner `for st in sensor_types` loop. |
| 8 | **High** | `data-pipeline/generators/reading_generator.py` | 239–260 | **Outlier injection is ineffective.** After applying a random multiplier (`5.0`, `-4.0`, or `10.0`), the value is clamped to "reasonable bounds." For bounded sensors like `water_quality` (clamp 0–100) and `humidity` (clamp 0–100), the outlier is almost always erased. For example, `water_quality` base ≈78 × 10 = 780 → clamped to 100, which is indistinguishable from normal high readings. | Remove post-outlier clamping for the outlier step, or use a smaller additive offset instead of a multiplicative one. Alternatively, flag outliers in `metadata` instead of relying solely on the value. |
| 9 | **High** | `data-pipeline/mqtt/subscriber.py` | 27–49 | The `_on_message` callback performs a **synchronous blocking HTTP request** (`requests.post`) directly inside the MQTT client's network loop thread. If the backend is slow or rate-limited, the network loop stalls, PINGREQ/PINGRESP messages are delayed, and the broker disconnects the client—causing **message loss** and **cascading backpressure failures**. | Use a **thread-safe queue** (e.g., `queue.Queue`) inside `_on_message` and a separate worker thread (or `ThreadPoolExecutor`) to drain the queue and call the API asynchronously. |
| 10 | **High** | `data-pipeline/seed_demo.py` | 173–230 | **No idempotency or cleanup.** Re-running `seed_demo.py` creates a brand-new set of stations (new UUIDs) and readings every time. There is no check for existing demo stations, no deduplication, and no cleanup flag. The database accumulates duplicate demo data on each run. | Add a `--cleanup` / `--reset` flag that deletes existing stations/readings for the demo user before seeding, or derive station IDs from a deterministic hash (e.g., `email + city + name`) so re-runs are idempotent. |
| 11 | **High** | `data-pipeline/seed_demo.py` | 219–230 | **No retry or backoff on HTTP requests.** `register_user`, `login_user`, `create_station_api`, and `ingest_bulk` all use one-shot `requests.post` with no retry logic. | 12 | **High** | `data-pipeline/seed_demo.py` | 182–193 | **Cascading failure from tier limit.** When `create_station_api` fails with 403 (station limit), the script falls back to the local station dict and appends it to `created_stations`. Do not add locally-fallback stations to the ingest list. |
| 13 | **High** | `data-pipeline/mqtt/publisher.py` | 44 | `mqtt.Client(client_id=...)` is incompatible with **paho-mqtt v2.x**. The constructor now requires a `callback_api_version` positional argument (e.g., `CallbackAPIVersion.VERSION1`). `requirements.txt` allows `>=1.6.1`, which resolves to v2 on fresh installs, causing an `TypeError` on import. | Pin the dependency to `paho-mqtt>=1.6.1,<2.0.0` in `requirements.txt`, or upgrade the code to use the paho-mqtt v2 API. |
| 14 | **High** | `data-pipeline/mqtt/subscriber.py` | 81 | Same paho-mqtt v2 constructor incompatibility as Issue #13. | Same fix as Issue #13. |
| 15 | **High** | `data-pipeline/seed_demo.py` | 219–230 | **Rate-limit violation.** With `BATCH_DELAY_SECONDS = 0.2`, the seeder submits ~5 batches/second = **300 requests/minute**. For free/pro tiers, the script is guaranteed to be rate-limited. | Make `BATCH_DELAY_SECONDS` configurable via CLI and dynamically adapt it based on the user's tier (e.g., read tier from `/api/v1/auth/me` and compute a safe delay). |

## Summary

The data pipeline has **six critical issues** that make the demo seeder and MQTT subscriber **non-functional** against the current backend API:

1. **Demo password fails backend validation** → registration is impossible on first run.
2. **Demo tier is ignored on registration** → user is free tier, so only 1 of 30 stations can be created.
3. **Station creation sends wrong field names** → all stations are created with `latitude=null` and `longitude=null`.
4. **Ingest endpoint receives raw list instead of `{"readings": [...]}`** → every batch fails with 422.
5. **MQTT subscriber sends no auth header** → every forwarded message fails with 401.
6. **MQTT subscriber also sends raw list to ingest** → even if auth were fixed, ingest would still fail with 422.

Beyond these blockers, there are **significant physics/logic bugs** in the reading generator (broken cross-sensor correlations, ineffective outlier injection), **performance and reliability issues** in the MQTT layer (blocking callbacks, no QoS-1 delivery confirmation, no retries), and **operational gaps** in the seeder (no idempotency, no cleanup, no rate-limit awareness, no graceful exit codes).

**Recommendation:** Do not use the current data pipeline for automated demos or CI until the Critical and High issues are resolved. The most impactful fixes are:
- Correct the password and tier-upgrade flow.
- Align the JSON payloads and field names with the backend schemas.
- Add authentication to the MQTT subscriber.
- Pre-compute correlated sensor values per timestep in the generator.
- Implement retry/backoff and idempotency in the seeder.
